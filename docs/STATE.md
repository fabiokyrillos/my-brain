## Status summary

Phase 1 is implemented as a hardened pre-MVP foundation. Sprint 1.5 remains closed. Phase 2A operational reliability remains deployed. Phase 2B is implemented and deployed through migration `023`: captures use the persisted lifecycle, interpretations are immutable snapshots selected by an owned current pointer, corrections and undo append versions, trust/entity evidence is deterministic and persisted per element, and synchronous reprocessing is protected by an expiring database lease. Desktop/mobile and remote behavior are verified. Phase 2X — Product Convergence has an approved architecture review, PRD, and implementation plan. Slice 2X.1 supplies pure daily-cycle contracts and guardrails. Slice 2X.2 is complete and deployed through migration `024`: a private, allowlisted, idempotent product-events ledger now has distinct user/server RPCs, generated schema types, a server-only best-effort boundary, a thin acknowledgement action, pgTAP coverage, and a disposable remote smoke. The historical product-projection commit `9f0c1e6` is preserved and reclassified as reusable prework. Slice 2X.3 is deployed through migration `025`: authenticated capture/reprocessing enqueue RPCs atomically persist only an entry plus a bounded `interpret_entry` job, while service-role claims use the existing leased queue. Slice 2X.4 is complete and deployed through migrations `026` and `027`: `interpret_entry` jobs are now actually processed end to end by a deployed worker (`supabase/functions/process-jobs/entry.ts`), reachable both by direct authenticated invocation and by an unattended per-minute `pg_cron`/`pg_net` dispatch drain authenticated by a Vault-backed secret; `process-jobs` was split into `index.ts` (auth/routing), `dispatch.ts` (fail-closed type router and drain loop), `attachment.ts` (unchanged attachment behavior, extracted), and `entry.ts` (new). Migration `027` additionally fixed a Slice 2X.3 regression that had been silently breaking every real file upload since migration `025` shipped (see ADR-022). The official Slice 2X.5 is complete with no new migration: `captureEntry` and `reprocessEntry` now call `capture_entry_async`/`enqueue_entry_reprocessing` directly and return as soon as that atomic persistence settles — no redirect, no synchronous AI call. A non-blocking `after()`-scheduled callback nudges the deployed worker and records best-effort product events; the automatic `pg_cron` drain remains the correctness backstop if the nudge fails. `agent/actions.ts` gained a generalized `retryProcessingJob` for entry jobs (no UI consumer yet — lands with the Needs-Attention slices). The now-unreachable synchronous Node extraction orchestrator (`src/features/interpretations/interpret-entry.ts`) was removed; the Deno worker remains the only production extraction path (see ADR-023). The official Slice 2X.6 is complete with no new migration: `src/features/daily-cycle/inbox-projection.ts` (`loadInboxProjection`) is the first production consumer of the Slice 2X.1 lifecycle mapper — it reads a page of owner-scoped `entries`, each entry's latest `interpret_entry` job, the current interpretation's `task_candidates`, open pending questions, and materialized tasks, then resolves each entry's `ProductState`/`AttentionReason` through `resolveDailyCycleLifecycle`. When that fail-closed mapper refuses an unrecognized internal combination, the loader still surfaces the entry as an explicit `could_not_organize`/`resolve_consistency` item instead of silently dropping it. `/inbox` and a new "Atividade recente" panel on Home both call this same projection and render the same new `InboxItemRow` component, so the two surfaces are guaranteed to agree on an entry's human-readable state; neither page computes lifecycle or reads `entries.status` directly anymore. `recordOnly`/`hasConsistencyIssue` are conservatively passed as `false` until Slice 2X.7 adds persisted candidate provenance — a documented, non-regression limitation, not a defect (see `CHANGELOG.md`). The official Slice 2X.7 is complete and deployed through migration `028`: `entry_interpretations.is_record_only` is now persisted (not just a transient input to the computed lifecycle status), `tasks` gained `source_interpretation_id`/`operation_key`, and a new `confirm_entry_task_candidates` RPC only ever materializes candidates belonging to the entry's actual current interpretation, refuses `record-only` interpretations outright, and is idempotent per operation key — replacing the old entry-wide `(source_entry_id, candidate_index)` uniqueness (which silently blocked a legitimate same-index candidate in a newer interpretation) with an interpretation-scoped one. `src/features/tasks/actions.ts` and `task-candidate-form.tsx` now bind every confirmation to the interpretation the UI actually displays; `src/features/interpretations/data.ts` exposes `isRecordOnly` per revision and a pure, tested `computeUnavailableCandidateIndexes` that conservatively treats provenance-less legacy tasks as unavailable. Exercising the new and existing confirmation RPCs as a real authenticated user against the linked project (not just via pgTAP or `service_role`) — done by actually running the new `scripts/remote-daily-cycle-smoke.mjs`, not only writing it — found and fixed two defects unrelated to this slice's own design: `confirm_entry_tasks` was `SECURITY INVOKER` and had never worked for a real user (missing `UPDATE`/`INSERT` grants on `entry_interpretations`/`undo_operations`/`audit_logs`; both `confirm_*` RPCs are now `SECURITY DEFINER`, matching every other RPC in this schema that writes to those tables), and any RPC raising SQLSTATE `40001` — including the already-shipped Phase 2B `correct_entry_interpretation` — hangs until gateway timeout on the linked project, a platform behavior confirmed via a raw `fetch()` with no application code involved. The new RPC signals its conflict with `55P03` instead; `correct_entry_interpretation`'s own equivalent `40001` path was deliberately left unchanged (out of scope for this slice's file list) and was tracked as an urgent, separate follow-up (see ADR-025, `TODO.md`, `SECURITY.md`). A standalone hotfix outside the slice sequence (migration `202607180029`, 2026-07-18) closed that follow-up: `correct_entry_interpretation` keeps its exact signature and every non-conflict behavior, now raising `55P03` instead of `40001` for its own version conflict, proven by an extended authenticated remote smoke to return in a bounded ~530ms with no gateway hang and no partial write (see ADR-026). The official Slice 2X.8 is complete with no new migration: `src/features/daily-cycle/review-projection.ts` and `src/features/daily-cycle/technical-details-projection.ts` are the two new server-only projection modules the entry-detail page now exclusively loads through — each pairs a pure, independently tested mapper (`toEntryReviewProjection`/`toEntryTechnicalDetailsView`) with a thin loader that reuses `loadInterpretationReview` (now internal infrastructure, no longer imported by any page). `src/app/[locale]/app/inbox/[entryId]/page.tsx` no longer reads `entries.status`, `entry.processing_error`, or any other raw Supabase row; its status badge, error/organizing banners, and retry-button visibility are driven entirely by the Slice 2X.1 `productState`/`availableActions` computed through the same `resolveDailyCycleLifecycle` mapper Caixa/Início already use (Slice 2X.6). A new architectural test (`page.architecture.test.ts`) enforces that boundary by forbidding `database.types`/`Database["public"]`/`@/lib/supabase/server`/`entry.status` in the page file. The two loaders run independently (each performs its own `loadInterpretationReview` call) specifically so a technical-details failure — malformed trust JSON, an unexpected query error — can never block or misreport the main review as ready; the page renders without the trust panel/comparisons section in that case, nothing else changes. `InterpretationRevisionEditor` and `TaskCandidateForm` are unchanged this slice and keep receiving the same prop shapes as before, now sourced from the review projection's non-frozen `editableCurrent`/`taskCandidates` fields rather than a page-level Supabase call. A standalone hotfix outside the slice sequence (2026-07-18, no new migration) resolved architecture-review finding F1: `hasMaterializedTaskForCandidates` was computed entry-wide ("any non-cancelled task exists for this entry") in both `inbox-projection.ts` and `review-projection.ts` instead of interpretation/candidate-scoped, so confirming only some of a multi-candidate interpretation's suggestions could make `productState` read `ready` while `TaskCandidateForm` still showed an unconfirmed candidate. Both loaders now derive that input from the same interpretation-scoped source `actionableCandidates` already used correctly (`computeUnavailableCandidateIndexes`), via a new shared pure helper `hasUnconfirmedTaskCandidates` (`src/features/interpretations/data.ts`); `lifecycle.ts` itself needed no change. See `docs/reports/PHASE_2X_CANDIDATE_LIFECYCLE_HOTFIX_REPORT.md`. The official Slice 2X.9 is complete with no new migration: the entry-detail page is now the decision-first, progressive-disclosure layout the PRD specifies, built entirely on the two Slice 2X.8 projections. `src/features/daily-cycle/entry-review.tsx` (`EntryReview`, `ReviewUnderstanding`, `ReviewAttention`, `ReviewNextActions`, `OriginalRecord`) renders, in order, what happened (`view.understanding` plus the `humanFields` DTO entries, rendered for the first time), what needs the user's decision (`view.attentionItems`, with the retry button and the specific error/pending-question text injected by the page as slot content rather than read by the component), the next action (candidate confirmation or the correction editor, now gated exclusively by `view.availableActions` instead of raw array lengths), and the always-available original record. `src/features/daily-cycle/technical-details.tsx` (`TechnicalDetails`) collapses the former two-column grid — trust/scores/policy/evidence per element, the immutable version history with field-by-field comparisons, and the structured extraction (concepts, dates, entity links, mentions) — behind a single native `<details>`, open on user request, unaffected by and independent from a technical-details load failure (still rendered inert with a fallback message, matching the Slice 2X.8 guarantee). `InterpretationRevisionEditor` gained an optional `showSummary` prop (default `true`) so the collapsed correction-editor state does not repeat the summary already shown in the understanding block; `TaskCandidateForm` needed no behavior change. No lifecycle, candidate-confirmation, or RPC logic changed — this slice is presentation/composition only over already-existing DTOs.
