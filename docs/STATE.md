Slice 2D.3 (Suggested Answers and Source/Effect Preview) is implemented and gate-verified on branch `codex/phase-2d-slice-3` (base `400ce03`, Slice 2D.2 merged via PR #12). It is a read-only slice: no resolution RPC version was added, `resolve_pending_question_v1`/`_v2` are untouched and byte-identical, and the closed `p_resolution` write shape is unchanged (ADR-033 reserves `_v3` for Slice 2D.4's consequence). A new pure `src/features/agent/question-suggestions.ts` derives bounded, closed, deduplicated suggested answers from the question's own leading interrogative plus the entry's own owned domain context — with no AI call, extraction-schema field, worker, provider, or Edge Function change. Because the extraction contract stores only `question`/`reason`/`confidence` (there is no persisted question type), the discovered taxonomy is `yes_no` (polar → the closed `{Sim,Não}`/`{Yes,No}` set), `person`, `project`, `organization`, and `context` (each drawn from the interpretation's own extracted entity names); `quando`/`when`, `por que`/`why`, and `quanto`/`how much` have no truthful deterministic answer in the current schema and correctly yield no chips, as does any supported kind whose owned context is empty — values are never fabricated. Options are bounded (≤6, ≤160 chars), deduplicated, dropped rather than truncated, ordered deterministically, and identified by semantics (`person:ana-prado`, `yes_no:yes`) so ids stay locale-independent while yes/no values localize. A new `server-only` `question-preview-projection.ts` returns a closed source DTO (question, reason, candidate index, bounded entry excerpt, entry dates, interpretation version/date/summary, truthful current/superseded indicator) after verifying question/entry/interpretation ownership and interpretation-to-entry provenance — no raw row, raw interpretation JSON, provider output, embedding, operation key, or audit internal reaches the client, and a cross-owner question is indistinguishable from a missing one. Its deterministic `toQuestionEffectPreview` returns the closed `{kind: "none" | "reinterpret", title, description, notice, willMutate: false}`, describing reinterpretation only as what a later confirmation could apply and always stating that nothing has been applied yet; nothing is selectable, enqueued, or mutated. Selecting a suggestion fills the existing editable answer field, moves focus there, and never submits; the browser may submit only a bounded suggestion id, which the Server Action authenticates by regenerating the deterministic options server-side and requiring the id to have been presented for that question and to carry exactly the submitted answer — a forged, stale, or edited-away id records `typed`, so attribution can never be forged and a UI hint never fails a resolution. The authenticated result is recorded as the bounded `origin: typed|suggested` enum on the persisted-outcome `question_answered_basic` event (the granularity PRD §18 sanctions), and additive migration `202607230049` extends only the product-event allowlist: the new property-free `question_effect_previewed`, the optional bounded `origin` property (optional so the pre-cutover application's `{}` payload keeps recording, making the migration independently deployable and the application commit rollback-safe), and a new truthful `questions` surface. The migration is applied to the linked development project at local/remote parity through `202607230049`; generated types are content-identical and unmodified. The branch is not pushed, has no PR, and no application code is deployed. Phase 2C is complete through Slice 2C.6 (merged via PR #10); the Phase 2D design is approved on `main`.
